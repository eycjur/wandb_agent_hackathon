# 検証設計

## 1. 目的

`llm-as-a-judge-mvp` の `self_pr` ドメインに絞り、以下の 2 つの仮説を検証する。

1. 生成改善仮説  
   人間が詳細なルールを追加で書かなくても、生成ログと評価ログから「良い自己PR」の特徴を抽出し、生成品質を改善できる。
2. Judge 改善仮説  
   人間評価ログから、Judge の評価を人間の感覚に近づけられる。

この検証では、単発の生成品質ではなく、`ログ収集 -> 改善 -> 未使用ケースで再評価` の流れで改善の再現性を確認する。

## 2. スコープ

- 対象ドメイン: `self_pr` のみ
- 対象機能:
  - 生成
  - Judge 評価
  - 人間評価
  - Judge プロンプト改善
  - Target プロンプト改善
- 対象外:
  - `resume_summary`
  - `resume_detail`
  - モデル比較そのもの
  - UI 改修

## 3. アプリ前提

`self_pr` ドメインでは、生成プロンプトと Judge ルーブリックがすでに定義されている。

### 3.1 生成の前提

- 200〜400 文字程度
- 2〜3 個の強みを簡潔にまとめる
- 実績ベースで、具体的な数値や根拠を含める
- 応募職種で活かせる専門性を 1 つに絞って強調する
- 事実に基づき、誇張しない

### 3.2 Judge の評価観点

- 実績・スキルが具体的な数値や根拠で示されているか
- 応募職種で活かせる専門性が明確に伝わるか
- 根拠のない曖昧な表現がなく説得力があるか
- 200〜400 文字で簡潔にまとまっているか
- 冗長でなく、採用担当の興味を引く内容か

### 3.3 人間評価で使える入力

アプリで保存できる人間評価は以下。

- `humanScore` : 0〜5
- `humanComment` : 任意コメント

Judge 改善には、同じ生成結果に対する `Judge 評価 + 人間評価` の組が必要になる。

### 3.4 応募職種の扱い

このアプリには応募先職種を明示的に入力する欄がない。  
そのため本検証では、各ケースを「入力文から最も自然に読める主軸職種向けの自己PR」として評価する。

ただし、入力文の末尾などに「今後は〇〇職を目指したい」のような志望職種が明示されている場合は、その志望職種を評価基準としてよい。

## 4. 検証で答える問い

### 4.1 Target 改善の問い

- 改善後の自己PRは、人間が見てより応募に使える内容になったか
- 改善後の自己PRは、専門性の絞り込みと具体性が向上したか
- 改善後の自己PRは、未使用ケースでも改善するか

### 4.2 Judge 改善の問い

- 改善後の Judge スコアは、人間スコアに近づいたか
- 改善後の Judge reason は、人間が問題視した点をより説明できているか
- Judge が「数字があるから高得点」のような単純化から脱し、自己PRの説得力や職種適合性を捉えられるか

## 5. データ設計

### 5.1 ケース数

初回検証では、合計 14 ケースを用意する。

- 学習用: 10 ケース
- 検証用: 4 ケース

学習用は改善案の生成に使い、検証用は改善に使わず holdout として残す。  
学習用を 10 ケースにする理由は、改善 API / UI が 10 件前後のログ利用を前提にしているため。

### 5.2 ケースの役割

| ID | 役割 | 狙い |
|---|---|---|
| SP-01 | 良質な定量実績あり | 基準ケース |
| SP-02 | 非エンジニア高実績 | 職種依存を弱める |
| SP-03 | PM/企画系 | 専門性の絞り込み確認 |
| SP-04 | CS/営業企画系 | 技術以外の自己PR確認 |
| SP-05 | 数字が少ない | 根拠不足ケース |
| SP-06 | キャリアチェンジ | 応募職種との接続確認 |
| SP-07 | 管理職中心 | 個人成果偏重の検出 |
| SP-08 | 曖昧表現が多い | 誇張・抽象表現耐性 |
| SP-09 | 若手で経験が浅い | 情報量不足ケース |
| SP-10 | 複数社短期在籍 | 強みの一貫性確認 |
| SP-11 | 英語・略語混在 | ノイズ耐性確認 |
| SP-12 | 守秘で詳細不足 | 捏造せず魅力化できるか確認 |
| SP-13 | 副業・兼務あり | 強みを1本に絞れるか確認 |
| SP-14 | 直近職種と過去の強みがずれる | どの専門性を前面に出すか確認 |

### 5.3 学習用 / 検証用の分け方

- 学習用: `SP-01` から `SP-10`
- 検証用: `SP-11` から `SP-14`

### 5.4 各ケースに必須で含める要素

- 現職または主な職種
- 主な役割
- 実績または成果
- 可能なら数値
- 応募先で活かせそうな専門性
- 一部のケースでは曖昧表現や不足情報もあえて含める

## 6. 人間評価設計

### 6.1 評価の基本方針

人間は Judge のスコアを見る前に、生成された自己PRだけを見て評価する。  
Judge を先に見ると、人間評価が引っ張られて Judge 改善の教師信号が弱くなるため。

### 6.2 必須評価項目

各生成結果に対して、最低限以下を記録する。

| 項目 | 必須 | 内容 |
|---|---|---|
| `case_id` | 必須 | SP-01 など |
| `humanScore` | 必須 | 0〜5 |
| `ready_to_use` | 必須 | 応募書類にそのまま使えるか |
| `judge_agreement` | 必須 | `agree / judge_high / judge_low` |
| `issue_tag_1` | 必須 | 最大の問題 |
| `issue_tag_2` | 任意 | 次点の問題 |
| `good_point_tag` | 任意 | 良い点 |
| `comment` | 推奨 | コメント |
| `elapsed_sec` | 推奨 | 評価時間 |

### 6.3 スコア基準

| スコア | 解釈 |
|---|---|
| 5 | そのまま応募に使える高品質な自己PR |
| 4 | 実務利用可能だが軽微な改善余地あり |
| 3 | 要点はあるが説得力または具体性が弱い |
| 2 | 曖昧で根拠不足が目立つ |
| 1 | 専門性や強みがほぼ伝わらない |
| 0 | 自己PRとして成立していない |

### 6.4 コメント記入ルール

以下のどちらかに当てはまる場合はコメントを必須にする。

- `humanScore <= 3`
- `|humanScore - judgeScore| >= 2`

コメントは自由文ではなく、次の型に揃える。

```text
良い点:
不足:
最優先で直す点:
```

### 6.5 issue_tag 候補

- `数字不足`
- `専門性がぼやける`
- `強みが多すぎる`
- `強みが弱い`
- `応募職種との接続が弱い`
- `誇張気味`
- `根拠不足`
- `冗長`
- `文字数不足`
- `文字数超過`
- `印象が弱い`
- `説得力あり`
- `具体性が高い`
- `簡潔`

## 7. 実験手順

### 7.1 Phase 0: 準備

- `self_pr` 用の学習用 10 ケースと検証用 4 ケースを作成する
- ケース ID を固定する
- 人間評価用の記録シートを用意する
- W&B / Weave が有効な環境で実施する
- 学習用ログと検証用ログは別の保存先に分ける

推奨:

- 学習用プロジェクト: 改善用ログを蓄積する
- 検証用プロジェクト: holdout 比較専用に使う

同じ Weave プロジェクトに学習用と検証用を混ぜると、改善時に holdout が学習データへ混入する。

### 7.2 Phase 1: ベースライン収集

学習用 10 ケースについて以下を実行する。

1. `self_pr` で生成
2. 人間評価
3. Judge 評価
4. `judge_agreement` と必要なコメント追記

この時点で以下のベースラインを記録する。

- `humanScore` 平均
- `judgeScore` 平均
- `ready_to_use` 比率
- `平均絶対誤差 = mean(|humanScore - judgeScore|)`
- `issue_tag` 出現頻度

### 7.3 Phase 2: Judge 改善

学習用 10 ケースのログを使って Judge プロンプト改善を実施する。

推奨順:

1. `meta`
2. `fewshot` は必要に応じて追加

Judge 改善後は、同じ学習用 10 ケースの生成結果に対して再度 Judge を実行し、人間評価との差分を比較する。

### 7.4 Phase 3: Target 改善

学習用 10 ケースのうち、低スコアまたは不合格のケースを使って Target プロンプト改善を実施する。

推奨順:

1. `meta`
2. `fewshot` は必要に応じて追加

Target 改善後は、学習用 10 ケースを再生成し、Judge 評価と人間評価を再取得する。

### 7.5 Phase 4: 検証用ケースでの再評価

検証用 4 ケースについて以下を実行する。

1. 検証用プロジェクトで、改善前プロンプトの結果を記録する
2. 同じ検証用ケースに対して、改善後プロンプトの結果を記録する
3. 人間評価で比較する

ここでは、改善に使っていない入力で改善が再現するかを確認する。  
検証用ケースのログは、Judge 改善 / Target 改善の入力に使わない。

### 7.6 Phase 5: 判定

以下をもとに、仮説が支持されるかを判断する。

- 検証用ケースで `humanScore` が改善したか
- 検証用ケースで `ready_to_use` 比率が改善したか
- 検証用ケースで `mean(|humanScore - judgeScore|)` が改善したか
- issue_tag の中でも主要な問題が減ったか

## 8. 成功条件

初回検証では、以下を成功条件の目安とする。

### 8.1 Target 改善の成功条件

- 検証用 4 ケースで `humanScore` 平均がベースラインより `0.5` 以上改善
- `ready_to_use = Yes` の件数が増加
- `数字不足`、`専門性がぼやける`、`冗長` の issue_tag が減少

### 8.2 Judge 改善の成功条件

- 検証用 4 ケースで `mean(|humanScore - judgeScore|)` がベースラインより改善
- `judge_high` と `judge_low` の件数が減少
- Judge の `reason` が人間コメントの主要指摘により近づく

### 8.3 仮説全体の成功条件

- 人間が新しい詳細ルールを追加しなくても、ログと評価だけで改善が確認できる
- 改善が学習用ケースだけでなく、検証用ケースでも確認できる

## 10. リスクと注意点

- ケース数が少ないため、数値は方向性を見るための指標として使う
- 評価者が 1 名だけだと主観の影響が強い
- Judge 改善と Target 改善を同時に変えると因果が見えにくい
- 改善前後の比較では、どのプロンプト版を使ったかを必ず残す
- Weave に保存されたログと、手元のケース ID を対応づける必要がある
- 学習用と検証用のログ保存先を分けないと、holdout が汚染される
