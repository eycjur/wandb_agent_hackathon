# 自己PR 人間評価ガイド

## 1. 目的

このガイドは、`self_pr` ドメインの生成結果を人間が一貫して評価するための手順書である。  
目的は、以下の 2 つの改善に使える評価データを安定して集めること。

1. Target 改善  
   生成された自己PRそのものの品質を改善する
2. Judge 改善  
   Judge のスコアと理由を人間の感覚に近づける

## 2. 評価の基本原則

- まず生成された自己PRだけを見る
- Judge のスコアと理由は、人間評価を付けた後に確認する
- 「自分なら応募書類に載せられるか」で判断する
- 完璧さではなく、実務利用可能性で評価する
- 入力にない事実を補って評価しない

## 3. 評価の対象

評価対象は、生成された `自己PR文` そのもの。  
元の職務経歴入力や Judge の評価結果そのものを採点するわけではない。

## 4. 評価の前に確認すること

各ケースは、以下の 2 つを見ながら評価する。

- [README.md](../cases/README.md)
  - `case_id`
  - `title`
  - `assumed_primary_role`
- `experiments/cases/inputs/train/*.txt` または `experiments/cases/inputs/eval/*.txt`
  - アプリへ貼り付ける入力本文

### 4.1 どの職種向け自己PRとして見るか

原則として、`assumed_primary_role` を基準に評価する。

ただし、入力文に以下のような明示的な志望がある場合は、その志望職種を優先してよい。

- `今後はプロダクトマネージャー職へキャリアチェンジしたい`
- `BizOpsや業務改善に近いポジションで強みを発揮したい`

### 4.2 評価の観点

以下の 5 観点で総合判断する。

1. 実績・スキルが具体的な数値や根拠で示されているか
2. 応募職種で活かせる専門性が明確に伝わるか
3. 根拠のない曖昧な表現がなく説得力があるか
4. 200〜400 文字で簡潔にまとまっているか
5. 冗長でなく、採用担当の興味を引く内容か

## 5. 評価手順

1. [README.md](../cases/README.md) で `case_id` と `assumed_primary_role` を確認する
2. 対応する `.txt` を開く
3. `.txt` の本文をアプリへ貼り付けて生成する
4. 元の入力文をざっと読む
5. 生成された自己PRを読む
6. `humanScore` を付ける
7. `ready_to_use` を判定する
8. `issue_tag_1` を付ける
9. 必要なら `issue_tag_2` と `good_point_tag` を付ける
10. コメントを書く
11. その後で Judge 結果を見て `judge_agreement` を付ける

### 5.1 どのファイルを開くか

- 学習用ケース:
  - `experiments/cases/inputs/train/SP-01.txt` のようなファイル
- 検証用ケース:
  - `experiments/cases/inputs/eval/SP-11.txt` のようなファイル

ケースの本文そのものは `.txt` が正本であり、評価時はこの本文をそのまま使う。

## 6. スコア基準

| スコア | 判断基準                                                               |
| ------ | ---------------------------------------------------------------------- |
| 5      | そのまま応募に使える。強みが明確で、具体性・説得力・簡潔さが揃っている |
| 4      | 実務利用可能。軽微な改善余地はあるが、応募書類として十分使える         |
| 3      | 大枠は使えるが弱い。具体性、職種適合、印象のいずれかが不足している     |
| 2      | 使いにくい。曖昧表現や根拠不足が目立ち、訴求力が弱い                   |
| 1      | 強みや専門性がほぼ伝わらない。自己PRとして成立が弱い                   |
| 0      | 自己PRとして成立していない。事実誤認、意味不明、過度な冗長などがある   |

## 7. `ready_to_use` の基準

### `Yes`

- 大きな書き直しなしで応募書類に載せられる
- 修正しても語尾や一部表現の調整程度

### `No`

- 強みの立て方から見直しが必要
- 数字や根拠が弱く、説得力が不足している
- 応募職種との接続が弱い
- 冗長または曖昧で、そのままでは使いにくい

## 8. `judge_agreement` の付け方

Judge のスコアを確認した後に、以下のいずれかを記録する。

### `agree`

- Judge のスコア感と人間のスコア感が概ね一致している
- 理由も大きくずれていない

### `judge_high`

- Judge が人間より高く評価しすぎている
- 例:
  - 数字が少ないのに高評価
  - 曖昧表現が多いのに高評価
  - 職種適合が弱いのに高評価

### `judge_low`

- Judge が人間より低く評価しすぎている
- 例:
  - 数字が少なくても十分に説得力がある
  - 管理職や企画職の強みを適切に拾えていない
  - 守秘制約下で妥当な表現を低く見積もっている

## 9. issue tag の使い方

### 9.1 `issue_tag_1`

最も大きい問題を 1 つ選ぶ。

### 9.2 `issue_tag_2`

問題が複数ある場合のみ選ぶ。

### 9.3 `good_point_tag`

良かった点が明確な場合のみ選ぶ。

## 10. issue tag 一覧

### 問題タグ

- `数字不足`
  - 成果はあるが数値や根拠が薄い
- `専門性がぼやける`
  - 何の職種として売りたいのかが弱い
- `強みが多すぎる`
  - アピール点が散っていて焦点が定まらない
- `強みが弱い`
  - 強みとして出している内容が平凡、または印象に残らない
- `応募職種との接続が弱い`
  - 経験はあるが、応募職種でどう活きるかが見えにくい
- `誇張気味`
  - 入力に対して断定が強すぎる、盛って見える
- `根拠不足`
  - 実績や主張に裏付けが少ない
- `冗長`
  - 長い、同じことを繰り返している、読後のキレが弱い
- `文字数不足`
  - 短すぎて訴求が足りない
- `文字数超過`
  - 長すぎて自己PRとして扱いづらい
- `印象が弱い`
  - 大きな欠陥はないが採用担当の記憶に残りにくい
- `事実飛躍`
  - 入力から言い切れない内容まで主張している

### 良い点タグ

- `説得力あり`
- `具体性が高い`
- `簡潔`
- `職種適合が明確`
- `印象が強い`

## 11. コメント記入ルール

以下に当てはまる場合、コメントは必須。

- `humanScore <= 3`
- `|humanScore - judgeScore| >= 2`

それ以外でも、後で改善案に使えそうな気づきがあればコメントを書く。

コメントは次の型を使う。

```text
良い点:
不足:
最優先で直す点:
```

### 良いコメント例

```text
良い点: 顧客課題を整理して改善につなげた経験が読み取りやすい。
不足: PM向け自己PRとしては、要件定義や意思決定の具体例が弱い。
最優先で直す点: 営業経験ではなく、PMに転用できる課題設定力を主軸に絞る。
```

```text
良い点: 数字が入り、成果の大きさが直感的に伝わる。
不足: 強みが3つ以上並び、何を一番アピールしたいのかがぼやける。
最優先で直す点: 一番刺さる専門性を1つに絞る。
```

### 悪いコメント例

```text
なんとなく弱い
```

```text
よかったです
```

## 12. 特に注意して見るケース

### キャリアチェンジ系

- 過去職種の説明で終わっていないか
- 応募職種に転用できる強みへ変換できているか

### 管理職系

- 個人実装量の少なさだけで低く評価しない
- 組織改善や育成の価値を適切に見ているか

### 数字が少ないケース

- 数字がないだけで一律に低評価にしない
- 具体的な行動や構造化された説明があれば一定の評価は可能

### 守秘制約ケース

- 書けない情報の存在を前提にする
- 守秘のために抽象的でも、役割や難易度が十分伝わるなら過小評価しない

### 英語・略語混在ケース

- 略語の多さだけで低評価にしない
- 一方で、日本語としての読みやすさや応募書類としての整いは確認する

## 13. よくある迷いと判断方針

### Q. 数字が少ないが文章はうまい。何点にすべきか

数字不足はマイナスだが、職種適合と説得力があれば `3` や `4` はあり得る。  
数値の有無だけで機械的に決めない。

### Q. 入力が曖昧なので、出力も曖昧になった。低評価にすべきか

入力制約の影響は考慮するが、自己PRとして使えるかは別。  
使いにくければ率直に低めに評価してよい。

### Q. 文字数は適正だが、印象が弱い

`印象が弱い` または `強みが弱い` を付ける。  
総合では `3` 前後になりやすい。

### Q. Judge の理由は良いがスコアだけずれている

`judge_high` または `judge_low` を付けてよい。  
スコア感と理由のどちらがずれているかをコメントに書くと改善に使いやすい。

## 14. 最終チェック

提出前に次を確認する。

- `humanScore` を記入した
- `ready_to_use` を記入した
- `issue_tag_1` を付けた
- 必要時にコメントを書いた
- Judge 確認後に `judge_agreement` を付けた
